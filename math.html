0 · Executive summary & the cost hierarchy

One structural fact drives everything: the array+pos[] tour representation forces an O(n) operation on the hot path that nothing else needs. Every other cost is O(n·width) or O(n) once per trial; the move-application rebuild is O(n) per accepted move, ~1,700 times per trial at rl11849. That single asymptotic mismatch is ~50–65 % of wall-clock and is the whole reason the <15 s target is out of reach today.

The three highest-value moves (all keep LK intact)

  1. bit-identical big build R1 — Two-level doubly-linked tour list. Reversal O(n)→O(√n) (≈109× fewer touches at rl11849), between stays O(1). Faithful ADT ⇒ identical LK decisions ⇒ identical tours. ~2–3× alone.
  2. bit-identical arch R2 — Store candidate-edge distances next to the candidate ids. Kills most of the 1.85M/trial distance recomputes in the LK inner loop. Cheap, safe. ~1.2–1.5×.
  3. bit-identical R5 — Admissible α-gain pruning in the recursion. Use the 1-tree bound to drop branches that provably can't close positively → fewer LK nodes, same moves. ~1.2–1.5×.

Stacked, R1·R2·R5 ≈ 3–5× at n≈12k (rl11849 ≈140 s → ≈30–45 s). The remaining gap to 15 s is convergence/trial-count and parallelism (item 9), which are outside "pure math." Below, every step gets the three-lens treatment.

1 · The objective & cost model

We minimise, over permutations σ ∈ Sn,

L(σ) = Σi=0 n−1 d(σi, σi+1),    d(p,q) = ⌊√( (xp−xq)² + (yp−yq)² )⌉

Pure-math view

The rounding makes d a genuine (integer) metric, so the triangle inequality holds and 2-opt/3-opt gains are well-defined. One tempting micro-optimisation — comparing squared distances to skip the sqrt — is not valid here: the objective is the sum of rounded roots, and ⌊√a⌉ + ⌊√b⌉ is not monotone in a+b. Gains must use the rounded values. So the lever is not "avoid the sqrt in comparisons" but "compute each d once and never again" (→ §7, distance cache).

What a "trial" costs

A trial = construct/perturb → LK descent → (n≥1000) EAX merge → accept. The loop runs T = n·(1+ext) times. The per-trial cost is dominated by how many moves LK accepts (each pays B4) and how many nodes it explores (each pays B3). The item-1 counters (measured, cached, seed 12345) pin the ratio:

2 · B4 — Tour representation the dominant lever

Current state

The tour is the permutation array tour[] plus its inverse pos[] plus flat next[]/prev[]. Queries are O(1): between(a,b,c) compares positions, next/prev are array loads. The price is the update: a general accepted move re-traces the whole Hamiltonian cycle from tour[0] to re-linearise, then recomputes pos/next/prev — O(n) per move. A depth-2 2-opt has an O(segment) fast path (reverseSegment), but it only fires for one orientation and catches ~4,141 of ~698k moves; the rest pay full O(n).

Pure-math view

This is exactly the classic "dynamic ordered list supporting reversal" problem. A 2-opt move is a segment reversal of the cyclic sequence; the data-structure question is how to support {next, prev, between, reverse} with sublinear reversal. Fredman, Johnson, McGeoch & Ostheimer (1995) proved a cell-probe lower bound of Ω(log n / log log n) per operation for this class — so O(log n) structures are essentially optimal, and the array's Θ(n) reversal is provably wasteful. The reason the array can't do better: between needs a total order (the positions), and a reversal renumbers Θ(segment) positions.

Algorithmic view — the menu (FMG&O 1995, Johnson–McGeoch survey)

The two-level list (recommendation R1). Partition the tour into ≈√n segments, each a doubly-linked list of cities, each carrying a reversed bit and a segment id + sequence number; segments themselves form a doubly-linked list with their own order ids. Then:

  • next(c)/prev(c): O(1) — follow the city link, honouring the segment's reversed bit.
  • between(a,b,c): O(1) — compare (segment order id, within-segment sequence id), reversed-bit-aware.
  • reverse(a,b): O(√n) — split the ≤2 boundary segments, then flip the reversed bits and reverse the order of the whole segments in between (no per-city work for interior segments), and renumber ≈√n segment ids. Periodic rebalance keeps segment sizes in [½√n, 2√n].

Why it is bit-identical. The LK core only ever asks the tour for next/prev/between/reverse. If the two-level list answers those identically to the array (same successor, same orientation convention), the search makes the identical sequence of decisions → identical accepted moves → identical tours and lengths. The internal layout is invisible to the search. This is a pure speed optimisation, gated on bit-identical lengths.

Hidden snag to design around. A handful of call sites read pos[] as a raw integer (e.g. improve2Opt: j = pos[c]; if (j <= i+1)). Those encode an array-specific layout. They must be rewritten in terms of the ADT's order query (between / a segment-rank comparator) before R1 is bit-identical. This is the bulk of the care.

Architecture view

Encapsulate the tour behind a single ADT (the existing TourView union is the right seam — collapse the dead segment branch first, per Stage-1 follow-up, then add the two-level implementation as a third variant gated by n). The LK core takes the ADT by interface; nothing in searchAdded/searchRemoved changes. The flip_* item-1 counters already exist to verify the √n behaviour.

Medium alternative if R1 is too much at once bit-identical

R4 — extend the in-place fast path. Today only one 2-opt orientation is applied incrementally. Add (a) the complementary 2-opt orientation (reverse the other segment, choosing the shorter side) and (b) Or-opt relocations, each in O(segment). The catch is the layout-match: the general retrace re-canonicalises from tour[0], so an in-place edit only matches it when the touched segment excludes tour[0] and preserves the start direction — detect that condition and fall back to the general path otherwise. This converts most general moves to O(segment) without a new structure, but it is fiddly and strictly weaker than R1 (still Θ(n) worst case). Use it as a stepping-stone or if R1's risk is unacceptable.

Estimated payoff (R1). √11849 ≈ 109 ⇒ the dominant bucket's element-work drops ~100×. By Amdahl with move-apply ≈ 60 %: new ≈ 0.6/100 + 0.40 ≈ 0.43 → ~2.3×; the freed cache pressure and cheaper between push it toward 2–3×, and the win grows with n (√n vs n).

3 · B3 — Lin–Kernighan search cost

LK from t1: alternately remove a tour edge (t2i−1, t2i) and add a candidate edge (t2i, t2i+1), maintaining the running gain

gi = Σk≤i d(removedk) − Σk<i d(addedk),   recurse while gi > 0,   close when gi − d(t2i, t1) > 0

This is the engine the user wants kept — so the levers are fewer nodes and cheaper nodes, not a different search.

3a · Cheaper nodes — candidate-distance cache (R2) bit-identical arch

Problem

The inner loop calls dist.distance(even, candidate) for every candidate at every node — ~1.85M lookups/trial at rl11849, the bulk of the lookup column. The candidate rows are fixed for the whole run and their distances were already computed during the candidate build, yet they are recomputed (or matrix-loaded) every visit.

Fix

Store a parallel cand_dist[node·width + k] = d(node, candidatek), filled once at build time. The LK loop then reads a contiguous, L1/L2-resident u32 instead of a 562 MB-matrix DRAM load or a sqrt. The alpha[] array is already laid out exactly this way — add a third parallel array. Non-candidate lookups (tour edges d(t1,t2), closing edges d(even, t1)) remain, but the candidate-edge majority becomes nearly free. Bit-identical (same numbers, faster source).

This is also the "candidate-distance option half-exists" from item 6: with it, the full n² matrix can be dropped entirely (hot distances cached, cold ones on-the-fly) — memory scales and the hot path speeds up. Best of both.

3b · Fewer nodes — admissible α-gain pruning (R5) bit-identical

Pure-math view

The minimum-1-tree machinery already gives, for free, a lower bound on the cost of any edge incident to a node. Define the reduced cost cπ(i,j) = d(i,j) + πi + πj. The α-value α(i,j) = cπ(i,j) − β(i,j) ≥ 0 (β = the 1-tree path bottleneck) means cπ(i,j) ≥ β(i,j), and the smallest reduced cost incident to a node is bounded below by its cheapest candidate. So for a partial LK sequence with running gain gi, the best possible remaining close is bounded: if even the most optimistic admissible continuation cannot reach gi − (min feasible add + close) > 0, the subtree is dead.

Algorithmic view

Concretely: candidates are already α-sorted, so once a candidate's g_i − d(even,odd_next) ≤ 0 in a sorted scan, all later candidates at that node are also non-improving (their added edges are ≥). The code already has the positive-gain check but iterates the whole row; turning it into an early-break on the sorted row is the cleanest admissible prune. A second, deeper prune: maintain a running upper bound on achievable gain using the cheapest available reduced cost at the frontier node and cut when it falls below the current best — this is α used as a branch-and-bound heuristic, still exact (no accepted move is lost), so bit-identical.

Care: "bit-identical" only holds if the prune is provably admissible (never skips a move that the un-pruned search would have applied). The sorted-row early-break is safe because the gain is monotone in the sorted added-edge cost; the deeper B&B prune needs a correctness proof per move type before it can claim bit-identity. Otherwise it is approx and goes through the multi-seed gate.

3c · Search completeness — the mirror direction (Bentley)

Standard neighbour-list 2-opt is incomplete unless you search from both endpoints of a removed edge: an improving move with Y2 near X2 (but Y1 far from X1) is only found from the mirror direction. The current orderTourEdgeChoices tries both tour edges at t1, which covers part of this, but it is worth auditing that the descent is not missing the X2-anchored half — that is quality (fewer trials to converge), which at fixed trial budget is effectively speed. approx

Architecture view (B3)

Don't-look bits already exist as the active-node queue (lkActive). Two cheap wins: (i) the candidate rows + cand_dist + alpha should be interleaved in one cache line per node (AoS within a row) so a node's whole candidate set is one or two cache lines; (ii) the gain arithmetic is all i64 on small ints — fine, no change. The dominant B3 cost is memory traffic (candidate + distance reads), which R2 + interleaving directly attack.

Estimated payoff. R2: cuts a large fraction of the 20–30 % lookup bucket → ~1.2–1.5×. R5 early-break: reduces lk_nodes and the per-node candidate scan → ~1.2–1.5×, free and bit-identical.

4 · A2/A3 — Candidate generation & the 1-tree

Current state

α-nearness candidates (Helsgaun): subgradient ascent maximises the Held–Karp 1-tree lower bound w(π) = L(Tπ) − 2 Σiπi over node penalties π, 32 iterations, each a dense O(n²) Prim; then α-values via per-row MST-path-bottleneck BFS, O(n²) total. ~33·n² ≈ 4.6×10⁹ work at rl11849 — the big one-time block (~3–5 % of the run, but pure O(n²): a wall at 10⁵⁺).

Pure-math view — why this is the right measure

The reduced cost cπ leaves the optimal tour invariant while reshaping the 1-tree; w(π) is a valid lower bound for every π, and maximising it (a concave non-smooth program) tightens the bound. Helsgaun's empirical payoff: 70–80 % of optimal-tour edges already sit in the minimum 1-tree, so α-nearness is a far better candidate filter than raw nearest-distance. Keep it. The improvements below make it cheaper, not different in spirit.

Algorithmic view — kill the O(n²)

  • R3a — MST on the Delaunay graph, O(n log n). For planar points the Euclidean MST is a subgraph of the Delaunay triangulation (in fact EMST ⊆ RNG ⊆ Gabriel ⊆ Delaunay), and the Delaunay graph has only O(n) edges with average vertex degree < 6. So: build Delaunay once (O(n log n)), run heap-Prim/Kruskal on its O(n) edges (≈O(n log n)). This replaces the dense Θ(n²) Prim. Subtlety: the EMST⊆Delaunay theorem is for the un-penalised metric. The π-reweighted 1-tree can want non-Delaunay edges, so restricting the ascent's MST to the Delaunay (or k-nearest) subgraph is an approximation — exactly the standard LKH practice for large n (build candidates on a subgraph). On the geometric fixtures the penalty perturbation is small and candidate quality is preserved in practice, but it is approx (different candidates ⇒ different trajectory ⇒ multi-seed gate). The repo already has a (deprecated) CGAL Delaunay path to reuse.
  • R3b — POPMUSIC candidate generation, near-linear. Taillard & Helsgaun (2019): build several POPMUSIC sub-tours (divide the tour into fixed-size sub-paths, optimise each independently, merge), then take the union of their edges as each city's candidate set (sorted by α without recomputing π). This produces high-quality candidates in almost-linear time, even for non-geometric instances, and scales to millions of cities — it is how LKH does 10⁵⁺. It sidesteps the subgradient cost entirely. approx, big build; the right answer for the 100k+ tier (roadmap item 7), overkill below 2×10⁴.

Pure-math view — a tighter bound in fewer iterations

The subgradient step is the classic πi += step·(7 gi + 3 giprev)/10 with period-halving (first period n/2, step0≈1) — Helsgaun's tuned heuristic, where gi = degi − 2 is the degree deficit. Faster/tighter convergence is available from the optimisation literature: deflected subgradient (Camerini–Fratta–Maffioli), the volume algorithm, or a small bundle method reach a tighter w(π) in fewer 1-tree builds. Since each iteration is one MST, halving the iteration count (32→~16) at equal bound quality is a direct ~2× on the candidate build. approx (a different π ⇒ different α-order ⇒ different candidates). Modest absolute win (build is ~4 %), so low priority below 10⁵.

Architecture view

The build is one-time and already heavily optimised (the O(n⁴)→O(n²) BFS fix was the big one). For the current targets (n≤18512) it is not the bottleneck; for 100k+ it is the first wall (memory: the dense build also implies the n² matrix). R3a/R3b are the same lever as item 6's "drop the matrix" + item 7's "sparse candidates" — they unlock scale, not current-fixture speed.

5 · B7 — EAX recombination

Current state

At n≥1000 the combiner forms the symmetric-difference multigraph G = E(trial) △ E(ref), decomposes it into AB-cycles (edges alternating trial/ref), applies improving cycles, repairs sub-tours with cheapest-candidate 2-opt bridges, and re-optimises only around the changed boundaries. References come from the elite pool. This is a faithful but simplified EAX.

Pure-math view

G is an even multigraph (every vertex has equal A-degree and B-degree), so it decomposes into edge-disjoint AB-cycles — a cycle-space fact. The choice of which cycles to apply (the E-set) is the whole game. Applying one AB-cycle = a generalised k-opt move that the sequential LK cannot reach (it can move interleaved bundles atomically). So EAX is not a competitor to LK — it is a different neighbourhood the LK polish then cleans up. Keeping LK intact, the lever is the E-set strategy.

Algorithmic view — Nagata's E-set strategies (GA-EAX)

  • single (current ≈ this): one AB-cycle per offspring — fast, converges the population quickly.
  • R6 — block2: select AB-cycles so the intermediate solution has few sub-tours and the offspring inherits more of parent B's edges. This is the strategy that makes GA-EAX strong on large instances — it exchanges edges globally and escapes local optima the single strategy gets stuck in. This is a quality lever for pr1002/fl1577/rl11849, exactly the n≥1000 regime. approx

Diversity — the principled metric (ties to item 4)

GA-EAX replaces a parent by the best offspring under an edge-entropy measure, not tour length — this is what actually preserves population diversity at scale. Our round-19 item-4 experiment used pairwise symmetric-difference (a coarser proxy) and found no regime on the current fixtures; the entropy formulation is the textbook-correct version and is the thing to try once item 6 unblocks d18512 / long runs (the regime where diversity matters). Keep it filed with item 4. approx

Estimated payoff. Accuracy, not raw speed: block2 + entropy is how GA-EAX beats LK-based solvers on 10⁴–2×10⁵ instances without an LK core — so as a combiner feeding our LK it is pure upside on the big rows. At fixed trial budget, better recombination = fewer trials to the same quality = effective speed.

6 · B1/B2 — Construction, warm-up, kicks

Small share (2–5 %), but two notes:

  • Kick (double-bridge): a 4-opt perturbation; the std.mem.rotate is O(segment) and, under R1, becomes a few O(√n) splices. The kick itself is mathematically fine — it is the canonical ILS perturbation and the do-not-retry table confirms alternatives lose. No change.
  • Warm-up 2-opt/Or-opt: O(passes·n·width) with O(1) delta evaluation already. Under R1 the reverseSegment in improve2Opt also drops to O(√n). The pos[c] comparisons here are the same array-layout call sites R1 must abstract (§2 snag).
  • NN / guided construction: already O(n·width) via candidate rows. Fine.

7 · Architecture, memory & distance

Distance representation

Measured (round 19): at rl11849 the 562 MB matrix and pure on-the-fly are time-neutral — lookups are not the bottleneck. So the distance decision is about memory + the hot fraction, not headline speed:

  • R2 candidate-distance cache (§3a) makes the hot lookups L2-resident; combined with on-the-fly for cold pairs, the n² matrix can be dropped → d18512 (1.37 GB) and 100k+ become feasible with no speed loss. This is item 6 done right: a memory enabler that also speeds the hot path.
  • Storing distances as u32 in candidate rows is 4·n·width bytes ≈ 240 KB at rl11849 — trivially L2-friendly.

Memory layout

  • Candidate data is already row-major SoA (data[], alpha[]). Add cand_dist[] as a third parallel array, or interleave (id, α, dist) per candidate so a node's row is one cache line. The LK inner loop becomes a single sequential sweep per node.
  • next/prev/pos are three n-arrays touched together every move — under R1 they are replaced by the two-level segment records; keep a segment's cities contiguous so a reversal touches contiguous memory (cache-friendly, unlike today's pointer-chase retrace).
  • Stage-1 follow-up: fully delete the dead two-level debug segment index (gated off in release but still allocated); the real two-level list (R1) replaces it with a structure that is actually read.

Structural seam

The single most valuable architectural change is making the tour an ADT interface (next/prev/between/reverse/materialize) that the LK core consumes by reference. That decouples "the search" (kept intact, per the user) from "the representation" (swap array → two-level → k-level by n). Every speed lever in §2 then lands behind that seam without touching searchAdded/searchRemoved.

Suggested order & the honest ceiling

1) R2 then R5 — both bit-identical, low-risk, immediately stackable (~1.5–2× combined) and they also prepare the ground (a clean candidate-row layout, a clean recursion) for R1. 2) R1 — the structural prize, behind the ADT seam, with R4 as a fallback if the layout-matching for full R1 proves too risky in one step. 3) R3a / R6 — scale and quality, when d18512 / 100k+ / the big rows come into focus.

Reality check on rl11849 < 15 s. R1·R2·R5 ≈ 3–5× ⇒ ≈140 s → ≈30–45 s. The last 2–3× to 15 s is not in these per-move levers — it is fewer trials (better candidates/recombination converging sooner, i.e. R5/R6 quality) and, ultimately, multi-stream parallelism into one elite pool (roadmap item 9, ~cores×). The math levers here are necessary and get most of the way; the final factor is convergence + cores, which is deliberately outside "leave the LK core intact, no new algo."
