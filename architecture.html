0 · Verdict & scorecard

This is a strong research prototype that has earned a refactor pass. 1 · What's genuinely good (do not "refactor" these away)

  • The bit-identical methodology. Debug-gated invariants (debugTourIsValid, the current_length == scan assert) + the sub-1000 canary + multi-seed {12345,7,99} gating. This is the reason rounds 11–19 could make aggressive changes safely; most research code has nothing like it. It is your biggest architectural asset.
  • Shadow-incumbent merge. EAX/IPT products accumulate in a shadow incumbent and fold in after the loop, so the kick/restart trajectory stays bit-for-bit identical and merge gains are pure upside. That's an elegant separation of "deterministic search" from "opportunistic recombination."
  • Zero hot-loop allocation. SolverWorkspace (30 fields) pre-allocates every scratch buffer once, with errdefer teardown. The trial loop never touches the allocator. Correct.
  • Measurement-driven size gates. eax_min_dimension, guided_max_dimension, the segment threshold — each is a documented, benchmarked boundary, not a guess. The do-not-retry table is institutional memory most projects lack.
  • Clean problem/solver layering. root.zig exposes a minimal surface (solve, Problem, SolveResult); problem.zig/tsplib.zig/exact.zig are properly isolated. Keeping ATSP as a problem-layer transform so the core stays symmetric is the right call.

The findings

H1 · solver.zig is a 6,361-line monolith HIGH cleanup

Now: 6,361 LOC · 76 top-level defs · 28 inline tests · ≈9 subsystems Effort: Medium (mechanical)

One file holds the distance oracle, the α/1-tree candidate build, three constructions (NN, farthest, guided backbone), the tour representation (FlatTourView/SegmentTourView/TourView), the LK engine + 2-opt/Or-opt + move application, IPT merge, EAX merge, the elite pool, the freeze subsystem, the solve loop, and 28 tests. Cohesion is low: unrelated concerns share a namespace, every change diffs the same file, and a newcomer (or you in three months) must hold all of it at once.

Fix

Split along the seams that already exist (the handoff lists this as deferred cleanup):

ModuleContents
distance.zigDistanceOracle (+ the candidate-distance cache, the penalty hook seam)
candidates.zig1-tree subgradient ascent, α-nearness, the Delaunay/POPMUSIC variants
tour.zigthe Tour ADT + array impl (+ future two-level impl) — see H2
search.zigLocalSearch: LK recursion, 2-opt/Or-opt, move application
recombine.zigIPT, EAX, elite pool (see M1)

Tests move next to their module. This is the enabling refactor: H2/H3/M1 are all easier once the boundaries are physical, not conceptual. Do it as pure code-motion first (zero behaviour change, bit-identical), commit, then start touching internals.

H2 · The tour ADT is only half applied HIGH perf-blocker

Now: TourView union exists, but LocalSearch also reads pos[]/next[]/prev[] directly Blocks: math.html R1 (two-level list, the ~2–3× lever)

You have a tour abstraction (TourView with next/prev/between/applyEdges) — but it leaks. LocalSearch carries the raw next/prev/pos slices as fields and indexes them directly; improve2Opt does j = self.pos[c]; if (j <= i+1); applyDepth2ClosingMove reads pos[b]/pos[d] as raw integers. These encode the array layout into the search logic.

This is the architectural reason the dominant perf lever is hard: you cannot swap array → two-level list while call sites assume "position is a dense integer I can compare with <=." It is both a boundary smell and the concrete blocker for the single biggest speedup.

Fix

  • Define one interface: next(c) · prev(c) · order(a,b,c) [=between] · reverse(a,b) · sequenceId(c) · materialize(out). sequenceId replaces raw pos[] reads with an opaque comparable token (the array impl returns the position; the two-level impl returns (segment-rank, in-segment-rank)).
  • Route every tour access in LocalSearch/2-opt/Or-opt/move-apply through it. The 47-field struct loses the raw next/prev/pos slices (it holds a *Tour instead).
  • Then the array impl and a two-level impl are interchangeable behind the seam, gated by n — exactly math.html R1, now unblocked.
Doing the seam first as a no-op refactor (array impl only, bit-identical) de-risks R1 entirely: you prove the abstraction is faithful before adding the fast structure underneath it.

✓ Done — LK-core seam landed (commit 5d2608b), bit-identical

The LK core now reaches the tour only through tourNext / tourPrev / tourSeq (opaque order token) + a node-based reverseBetween; the segment queries (between / circularSegmentSize / nodeInCircularSegment / segmentIsNoMoreThanHalf) were already ADT-shaped. Every bench row matches round-19 at 3 seeds, 44 tests both modes.

Coupling it surfaced (the point of doing it first): next[]/prev[] is a cache refreshed only by rebuildState. The warm-up sweeps mutate via reverseSegment (tour+pos only), so the cache is stale there — they still read successors from tour[]; only the pos-based seam (tourSeq/reverseBetween) is valid in the sweeps. R1's remaining work is exactly this: the two-level list must keep next/prev always-current, or the sweeps must iterate via the ADT. The no-op refactor found the hidden coupling before the risky structural swap — which is what it was for.

H3 · LocalSearch is a 47-field god-struct, hand-assembled HIGH

Now: 47 fields · built at 2 sites · copied for merge (var merge_search = search) twice Risk: the round-5 scratch-aliasing fragility

The solve loop constructs LocalSearch{ ... } with ~40 explicit field assignments, then later does var merge_search = search (a shallow copy that shares the workspace buffers). Three problems: (1) adding a field means editing every construction site and remembering the merge copy; (2) the shallow copy aliases scratch arrays — the round-5 note flagged that pos/next are stale between a merge win and the next trial's rebuild (currently harmless "because nothing reads them in that window," which is a latent-bug invariant held by comment, not by type); (3) config, scratch, and mutable state are intermixed in one flat struct, so it's impossible to tell at a glance what's input vs working memory.

Fix

  • Borrow the workspace by pointer: LocalSearch holds ws: *SolverWorkspace instead of 30 copied slices. Construction shrinks to a handful of fields; the merge copy stops aliasing because both reference the same owned workspace explicitly.
  • Split the remaining fields into three groups: Config (depths, passes, flags — immutable per trial), Tour (the ADT from H2), State (current_length, lk_active_*). A 47-field flat bag becomes three small, intention-revealing structs.

M1 · Two recombination engines (IPT + EAX) MED

Now: IPT below n=1000, EAX at/above — two code paths, two scratch structs (IptScratch + EaxScratch)

The handoff calls this "explicitly transitional until elite-pool replaces both," and mathematically EAX subsumes IPT (a section swap is one non-splitting AB-cycle). But — and this is the honest part — EAX below 1000 was measured strictly worse (it reshuffles knife-edge optima; +10% time, lost lin318/rd400/pcb442-class optima). So this is the one place where "specialised beats general" currently wins on the benchmark. The duplication is justified by data, not laziness.

Fix — but be honest about it

Don't keep calling it "transitional" if no one can make EAX reproduce the sub-1000 trajectories. Two real options: (a) invest in making EAX's non-splitting, smallest-section-first path bit-reproduce IPT below 1000, then delete IPT (true consolidation); or (b) accept it as a permanent, documented size-gated dual strategy and stop treating it as debt. Either is fine; the limbo ("temporary" for 9 rounds) is the smell. Under H1 both engines live in recombine.zig behind one recombine(trial, ref) → product entry point, which contains the size-gate in one place instead of scattering it through the solve loop.

M2 · ~31% of SolveOptions is a delete-candidate feature MED cleanup

Now: 29 option fields — ~9 are the freeze subsystem + diagnostic out-params

enable_edge_freeze, edge_freeze_{min_votes,fraction_x100,vote_mode,lk_respect,stale_window,soft}, inject_frozen, and frozen_edges_out are all item-3 (voting-freeze), which you already marked a DELETE CANDIDATE. That's roughly a third of the production options struct devoted to a proven-dead feature — plus frozen_edges_out is a *std.ArrayList diagnostic out-param living in the production options type, which is a layering smell (the public API surface carries measurement scaffolding).

Fix

  • Delete the freeze subsystem (the round-18 post-mortem authorises it): the ~9 option fields, the vote_node/vote_count workspace arrays, the vote primitives, the kick variant, the move guards, and the SolveStats freeze counters. This is the highest-ratio cleanup — it shrinks the options struct, the workspace, the stats, and the solve loop at once.
  • Move genuine diagnostics (a frozen/edge dump, injection probes) out of SolveOptions into a separate Diagnostics hook or keep them only in the profile driver. Production options should be production-only.

M3 · The cleanup backlog MED cleanup

Beyond freeze (M2), the deferred list the handoff already tracks:

  • Dead two-level segment index — gated off in release (Stage 1) but still allocated and maintained in debug; will be replaced by the real two-level list (H2/R1), so delete the placeholder when R1 lands.
  • Move-patching machinery — enable_move_patching defaults off; tryPatchTwoComponents + the patch counters carry weight for a path that isn't on.
  • farthestInsertionTour — still wired (cold-restart branch, trial%4==1 && n≥300) but a minor third construction; candidate for folding into the guided/NN path.
  • CGAL Delaunay — optional dependency with no measured benefit; you plan to deprecate. The build coupling (gmp/mpfr/c++) is real maintenance cost. However — note that math.html R3a wants a Delaunay graph for the O(n log n) MST, so before ripping CGAL out, decide whether a small pure-Zig Delaunay (or k-NN graph) replaces it for that purpose.
  • SolveStats: 68 counters — several (lk_chain_nonseq_depth_* arrays, parts of move_plan_*) are instrumentation from closed investigations. Prune in the same pass.

Rough estimate: M2+M3 together remove well over a thousand lines from the monolith and cut three struct field-counts — a large readability win for low risk (all default-off / dead paths).

M4 · No differential-test harness for representation swaps MED perf-blocker

Your invariant checks validate one representation against the tour array. But the way to land R1 (two-level list) safely is a differential harness: generate random tours + random valid move sequences, and assert the new structure and the array agree on next/prev/order/reverse at every step. You don't have that, and it's the cheapest insurance for the biggest change.

Fix

Add a property test in tour.zig's tests: for k random tours × m random moves, assert impl_new.query == impl_array.query for all query types. Build it before R1 — it turns "hope it's bit-identical" into "proven on 10⁶ random ops." This generalises: any future rep (k-level, splay) reuses the same harness.

L1–L4 · Lower-severity

L1 · Build & binary friction LOW cleanup

profile.zig is built ad-hoc via a zig build-exe line that needs a hand-written build_options stub (the with_cgal=false file in /tmp). And commiv-profile is a git-tracked binary that every build overwrites → permanent spurious M commiv-profile in status. Fix: add a profile step to build.zig (so it gets the real build_options module, no stub), and .gitignore the binary. Removes a recurring papercut.

L2 · Distance cap is weight-count, not cache-aware LOW

max_distance_cache_weights is a raw element count (default 4M). The decision it encodes is really "does the matrix fit a cache budget?" — a byte question (4M weights = 16 MB ≈ L3). Make the cap byte/L3-aware and self-documenting; pairs with the math.html R2 candidate-distance cache so the matrix can be dropped above the budget while hot lookups stay L2-resident. (Round-19 measured the matrix-vs-on-the-fly is time-neutral, so this is a memory/scaling cleanup, not a speed claim.)

L3 · SolveOptions mixes config tiers LOW

Even after freeze is deleted, the struct flatly mixes "algorithm shape" (candidate_count, lk_max_depth, candidate_mode) with "budget" (trials, extension factor) with "internal tuning" (backtrack limits, ascent iterations, patch min-gain). Grouping into nested sub-structs (Budget, Search, Candidates) makes the public surface legible and signals which knobs a caller should ever touch.

L4 · The VRP/HGS extension seams don't exist yet LOW extensibility

See §2 — the future layer needs a distance penalty hook and pinned-edge support; neither has a seam today. Low severity now (future scope), but cheap to define the interface early so the core doesn't have to be reopened later.

2 · Extensibility — the VRP/HGS seams

Your stated north star is an HGS-style VRP layer on top of this symmetric-TSP core (constraints via distance-matrix manipulation + penalty hooks; ATSP via the 2n transform at the problem layer). The architecture mostly supports this, which is to its credit. The gaps:

ConnectorStatusSeam needed
ATSP (2n Jonker–Volgenant transform)supportedproblem-layer only; core stays symmetric ✓
Elite pool / populationexistsalready built for EAX; the HGS population reuses it
Distance penalty hooks (constraints)missingDistanceOracle needs distance(a,b) = base(a,b) + penalty(a,b) with a mutable penalty source
Pinned / fixed edgesmissinga "this edge must stay" predicate the move-apply + kick respect (the freeze machinery was the wrong version of this — but a caller-supplied hard constraint is legitimate)
Split / route-extraction DPfuturea layer above solve(), not inside it — keep the core unaware
Diversity metric (edge entropy)partialitem-4 symdiff exists; the entropy version (math.html R6) is the HGS-correct one
Note on pinned edges vs. freeze. The freeze subsystem you're deleting was an internal heuristic trying to guess which edges to fix — proven a dead end. A VRP caller-supplied pinned-edge constraint is a different, legitimate thing (the user knows the constraint). When you build it, build it as a clean predicate on the move-apply path, not by resurrecting the vote machinery.

Defining the DistanceOracle penalty hook now (an optional penalty: ?*const PenaltySource, default null, added in distance()) costs almost nothing and means the VRP layer never has to reopen the core's hot path. That's the one forward-looking seam worth adding early.

The through-line

Steps 1–2 are pure hygiene (delete dead weight, then physically separate the live concerns) — both bit-identical, both make everything after them easier, and together they fix the cohesion grade (D→B) and the dead-weight grade (C→A) with near-zero risk. Steps 4–6 are the structural payoff: the tour-ADT seam is simultaneously the right boundary and the unlock for math.html's biggest speedup, and shrinking LocalSearch falls out of it. Do the cheap hygiene first; it clears the field for the high-value seam.

Bottom line: your instinct that the architecture is good is half-right — the decisions are good, and the validation discipline is genuinely excellent and worth protecting. What's missing is a hygiene pass and one real abstraction boundary (the tour). Neither requires changing the LK core or the algorithm; both are mechanical-to-medium effort; and the boundary you most need to draw is the same one that unblocks your biggest performance win.
